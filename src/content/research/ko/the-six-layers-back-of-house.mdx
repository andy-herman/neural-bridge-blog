개인 AI 시스템을 만들 때 대부분의 사람들은 대시보드부터 시작합니다. UI를 스케치하고, 프레임워크를 고르고, 칸반 컬럼 이름을 정하고, 다크 모드를 두고 갑론을박하죠.

저도 그럴 뻔했습니다. 그런데 제가 감탄했던 작동하는 개인 AI 데모들에는 하나같이 지루한 공통점이 있었습니다. 대시보드는 투박했지만, 그 아래의 상태 관리는 정교했다는 것이죠.

이것이 제가 어렵게 배운 교훈이자, 이제 추천하는 빌드 순서입니다. 6개 레이어, 아래부터 위로, 프론트엔드는 맨 마지막.

## 6개 레이어 소개

[Neural Bridge](/)는 독립적으로 교체 가능한 6개 레이어로 구성되어 있습니다:

| # | 레이어 | 이 빌드에서 구체적으로 |
|---|---|---|
| 1 | **에이전트** | `~/.claude/agents/*.md`의 역할별 설정 파일. YAML 프론트매터(description, tools, model, mcpServers, skills)를 포함하며, 각각 특화된 Claude Code 인스턴스입니다. |
| 2 | **스킬** | `~/.claude/skills/*`에 있는 재사용 가능한 기능들로, 모든 에이전트가 상속합니다 (pptx, docx, 웹 리서치, 공개적 개발 보이스 규칙, 코딩 원칙). |
| 3 | **전송** | 책상 밖에서 시스템과 소통하는 방법. Neural Bridge의 경우: Mac Mini 슈퍼바이저에 연결된 Discord 봇. |
| 4 | **공유 상태** | 임베딩을 위한 로컬 SQLite + sqlite-vec, 위키 레이어를 위한 마크다운 파일, 시크릿을 위한 macOS 키체인. 다른 모든 것이 읽고 쓰는 기반 시스템. |
| 5 | **오케스트레이션** | 앞선 4개 레이어를 연결하는 슈퍼바이저 프로세스. 저의 경우: `launchd`로 예약된, `claude -p` 서브프로세스를 생성하는 Python `discord.py` 데몬. |
| 6 | **프론트엔드** | 웹 대시보드, 탐색 가능한 칸반, 하이브 피드, 일정, 메모리 검색. |

각 레이어는 교체 가능합니다. Discord를 Telegram으로 바꾸면 3번 레이어만 바뀝니다. SQLite를 Postgres로 바꾸면 4번 레이어만 바뀌고, `discord.py`를 Node 봇으로 교체해도 5번 레이어만 영향받습니다. 레이어 간 인터페이스는 의도적으로 좁게 설계되어 있습니다.

## 빌드 순서: 1 → 2 → 3 → 4 → 5 → 6

순서를 건너뛰지 마세요. 프론트엔드는 첫 번째가 아니라 **마지막**입니다.

대부분의 사람들이 여기서 실수를 합니다. 충분히 짚고 넘어갈 가치가 있습니다.

대시보드는 상태 위에 얹힌 장식입니다. 4번 레이어가 탄탄하면 시각화는 사소한 문제입니다. 어떤 프레임워크를 골라도 깔끔한 데이터 위에서 깔끔한 보드를 렌더링해 줄 테니까요. 하지만 4번 레이어가 잘못되어 있으면, 어떤 프론트엔드도 구해주지 못합니다. 엉망인 구조 위에 예쁜 껍데기를 씌우는 것은, 잘 설계된 테이블 위에 단순한 태스크 로그를 올려두는 것보다 훨씬 나쁩니다. 대시보드가 썩은 부분을 가리기 때문입니다.

제 친구 중에는 덜 만들어진 스키마 위에 정교한 칸반 UI를 구축하고, 스키마를 중간에 다시 설계하고, UI가 마이그레이션을 흡수하도록 억지로 맞추는 과정을 겪은 사람이 여럿 있습니다. 정말 특별한 종류의 고통입니다.

레이어를 순서대로 쌓아가면, 이전 레이어를 직접 사용하면서 실제로 필요한 상태가 무엇인지 자연스럽게 알게 됩니다. 4번 레이어에 도달할 때쯤이면 3주간 없이 작업해봤기 때문에 스키마가 어떻게 생겨야 하는지 이미 알고 있습니다. 6번 레이어에 이르렀을 때는 남은 질문이 취향뿐입니다.

## 4번 레이어가 핵심인 이유

1번과 2번 레이어는 특화된 에이전트와 재사용 가능한 기능을 제공합니다. 3번과 5번 레이어는 이것들을 연결하고, 6번 레이어는 무슨 일이 일어나는지 볼 수 있게 해줍니다.

하지만 4번 레이어, 즉 **공유 상태**야말로 시스템을 *복리처럼 쌓이게* 만드는 핵심입니다. 지난 화요일에 어떤 주제를 조사했던 에이전트가 일요일에 다른 에이전트에게 유용할 수 있는 이유이고, 대화 기록을 검색할 수 있는 이유이며, 두 번째 작업 시간이 첫 번째 위에 쌓일 수 있는 이유입니다.

4번 레이어를 제대로 만들면 시스템은 사용할수록 점점 똑똑해집니다. 잘못 만들면 시스템은 매주 일요일 밤 11시 59분마다 모든 것을 잊습니다.

구체적으로, 4번 레이어는 다음을 담고 있습니다:

- 에이전트 간 공유되는 개념 아티클 위키, 즉 통합 지식 레이어
- 에이전트별 세션 기록과 일일 로그 (위키가 컴파일하는 원본 입력)
- 임베딩이 적용된 대화 기록 인덱스 (시맨틱 검색이 바로 작동하도록)
- 모든 항목의 출처 정보 메타데이터 (오염된 항목을 항상 추적할 수 있도록)
- 칸반 자체 (현재는 6단계 레이블 상태 머신을 가진 GitHub 이슈로 구성)

대시보드가 이 모든 것을 읽습니다. 에이전트들도 이 모든 것을 읽습니다. 오케스트레이터는 이를 기반으로 조율합니다. 4번 레이어는 단순한 데이터베이스가 아닙니다. 기반 시스템입니다.

## 대시보드 먼저라는 유혹

왜 모두가 대시보드를 먼저 만들고 싶어할까요? 대시보드가 *눈에 보이는* 유일한 레이어이기 때문입니다. 나머지는 모두 배관 작업이죠. 대시보드는 5분 안에 친구에게 보여주며 무엇을 만들고 있는지 이해시킬 수 있습니다. sqlite-vec 임베딩이 달린 SQLite 스키마를 보여줘서는 그렇게 할 수 없습니다.

하지만 '데모가 잘 된다'와 '잘 작동한다'는 다른 문제입니다. 대시보드는 시스템이 하는 일을 보여줄 뿐, 그 일을 직접 하지는 않습니다.

대시보드를 먼저 만든다는 것은, 아직 무엇이 필요한지 몰랐기 때문에 4번 레이어에 잘못된 추상화를 선택한다는 의미입니다. 그 추상화는 굳어집니다. 대시보드가 이미 그것과 연결되어 있어서 마이그레이션이 점점 비싸지거든요. 6개월이 지나면, 절반밖에 안 만들어진 상태 모델을 가리키는 아름다운 대시보드가 남고, 시스템은 더 이상 성장하지 못합니다.

처음부터 후회할 스키마 위에 완성된 대시보드를 올리는 것보다는, 3개월간 훌륭한 스키마 위에서 단순한 태스크 로그를 쓰다가 13주차에 대시보드를 추가하는 편이 낫습니다.

## 만든 것과 아직 못 만든 것

구체적으로 말하자면, 2026년 5월 중순 현재 Neural Bridge의 상태입니다:

- **1번 레이어 (에이전트):** 완료. Claude Code 플러그인에 9명의 전문가: research, teaching-prep, content, senior-pm, social, recruiter, automation-engineer, security-reviewer, docs-editor.
- **2번 레이어 (스킬):** 완료. 사용자 레벨 스킬 3개 (`hey-luna`, `goodbye`, `coding-discipline`)와 에이전트 내부 스킬 참조.
- **3번 레이어 (전송):** 완료. Discord 봇 데몬. 9개 에이전트 각각 고유한 Discord 정체성을 가지며, senior-pm이 슬래시 커맨드를 등록하고 나머지는 핸드오프 시 발언합니다.
- **4번 레이어 (상태):** 대부분 완료. 위키 + 일일 로그 + 게이트 검증 컴파일 파이프라인은 출시되었습니다. SQLite + 임베딩은 설계 단계이나 아직 이식하지 않았습니다 (코퍼스가 벡터 검색이 필요할 만큼 크지 않아서).
- **5번 레이어 (오케스트레이션):** 완료. Python 슈퍼바이저가 Mac Mini에서 `launchd` 사용자 에이전트로 실행 중이며, 크래시 시 재시작하고 키체인에 저장된 Discord 웹훅으로 메시지를 전송합니다.
- **6번 레이어 (프론트엔드):** 미완료. 현재 "프론트엔드"는 Discord와 GitHub 이슈 보드입니다. 웹 대시보드는 없습니다.

마지막 항목은 의도적입니다. 4번 레이어가 몇 주 더 쌓이면 실제로 무엇을 보고 싶은지 알 수 있고, 그때 6번 레이어에서 무엇을 만들지도 알게 됩니다.

## 직접 만든다면

시작하는 단계라면:

1. **에이전트부터 선택하세요.** 도움이 필요한 일 중에서 하나의 프롬프트로는 처리할 수 없을 만큼 톤이나 스킬이 다른 것이 3~5개 있나요? 각각이 에이전트가 됩니다.
2. **공유 스킬을 정의하세요.** 모든 에이전트가 접근할 수 있어야 하는 기능은 무엇인가요? 그것이 스킬입니다.
3. **전송 수단을 선택하세요.** Discord, Telegram, Slack, 터미널, 무엇이든 상관없습니다. 노트북을 여는 것보다 마찰이 적으면 됩니다.
4. **4번 레이어를 하나의 제품처럼 설계하세요.** 이 레이어가 복리로 쌓이는 레이어입니다. 시간을 투자하세요. [Karpathy의 LLM 지식 베이스 gist](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)나 [Cole Medin의 claude-memory-compiler](https://github.com/coleam00/claude-memory-compiler)를 참고하세요. 여기서 새로운 것을 발명하려 하지 마세요.
5. **작동하는 가장 단순한 오케스트레이터로 연결하세요.** Cron, launchd, 장시간 실행되는 Python 스크립트, TypeScript 데몬. 잠결에도 디버깅할 수 있는 것이면 됩니다.
6. **그 다음에, 마지막으로, 대시보드를.** 대시보드 없이도 시스템을 이미 운용할 수 있을 때, 운용을 더 편하게 만들어줄 시각화를 만드세요.

빌드 순서는 이데올로기가 아닙니다. 각 레이어가 다음 레이어에 무엇이 필요한지를 가르쳐주는 순서입니다.

## 이 프레임워크의 출처

6개 레이어 분해 방식은 제가 독창적으로 만든 것이 아닙니다. 개인 AI 빌더 커뮤니티에서 1년 넘게 회자되어 온 개념이죠. 제가 여기에 기여하는 것은 빌드 순서에 대한 주장입니다: **프론트엔드는 마지막, 4번 레이어가 핵심, 대시보드는 진행 상황에 대해 거짓말을 합니다.**

이 글에서 하나만 가져간다면 바로 이것입니다. 주방(back of house)을 먼저 만드세요. 홀(front of house)은 위에 얹힌 장식입니다.
